# Enterprise Audit Logging Guide

## Overview
The MT5 AI/ML Trading Bot implements a comprehensive audit logging system designed for regulatory compliance and post-incident forensic analysis. Every critical system decision is attributable, timestamped, and traceable.

## Audit Log Table Schema
The audit logs are stored in the `audit_log` table within the configured audit database (SQLite or PostgreSQL).

| Column | Type | Description |
|--------|------|-------------|
| id | Integer | Primary key, autoincrement |
| created_at | DateTime | UTC timestamp of the event |
| updated_at | DateTime | UTC timestamp of last update |
| actor | String | The system component or user performing the action |
| action | String | The specific action performed (e.g., `prediction`, `risk_decision`) |
| details | Text | Human-readable summary of the event |
| metadata_json | JSON | Structured context data (decision chains, model parameters, etc.) |

## Tracked Events

### 1. Configuration Snapshots
At every system startup, a sanitized snapshot of the active configuration is recorded.
- **Action:** `config_snapshot`
- **Metadata:** All non-sensitive configuration parameters.

### 2. Model Predictions
Every signal generated by the AI models is recorded before being passed to risk management.
- **Action:** `prediction`
- **Metadata:** Symbol, direction, confidence score, and algorithm-specific metadata.

### 3. Risk Engine Decision Chain
The full evaluation chain of the 6-layer risk filter cascade is recorded for every signal.
- **Action:** `risk_decision`
- **Metadata:** Results of each filter (circuit breaker, daily loss, etc.) and the final pass/fail status.

### 4. Blocked Trades
Whenever a trade is blocked by execution filters or capital allocation limits, the specific reason is logged.
- **Action:** `trade_blocked`
- **Metadata:** Specific filter that triggered the block and the associated context.

### 5. Deployment Events
Records successful system startups and version information.
- **Action:** `deployment`
- **Metadata:** Version number, environment (demo/live), and status.

### 6. Operator Actions
Manual overrides and system shutdowns are tracked.
- **Action:** `operator_shutdown`
- **Details:** Reason for the action (e.g., `KeyboardInterrupt`).

### 7. Technical Execution Filter Decision
The full technical trace of the 9-layer execution filter cascade is recorded.
- **Action:** `execution_decision`
- **Metadata:** Comprehensive metrics for each layer (ATR ratio, RSI value, EMA sequence status, etc.) providing institutional audibility for entry vetting.

## Forensic Querying
Audit logs can be queried directly via SQL for troubleshooting or compliance reviews:

```sql
-- Find why trades were blocked in the last 24 hours
SELECT created_at, details, metadata_json
FROM audit_log
WHERE action = 'trade_blocked'
  AND created_at > datetime('now', '-1 day');

-- Review the risk decision chain for a specific symbol
SELECT created_at, action, metadata_json
FROM audit_log
WHERE metadata_json ->> '$.symbol' = 'XAUUSD'
  AND action = 'risk_decision'
ORDER BY created_at DESC;
```
